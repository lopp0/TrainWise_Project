[2026-04-21] | kept intensityFactor in load formula | intensityFactor has been fully removed from the project (DB, backend, frontend) — session load = duration × exertion only
[2026-04-21] | edit-workout path in StatsScreen didn't send calculatedLoadForSession and didn't call calculateDailyLoad after save, so Warnings kept stale Red/high AC ratio | any screen that creates/updates an ActivityLog MUST set calculatedLoadForSession = duration × exertion AND call calculateDailyLoad(userId) afterwards — otherwise the DailyLoad row on the server won't reflect the edit
[2026-04-21] | calling calculateDailyLoad() only recalcs ONE day (default today) — editing an older session leaves that older day's DailyLoad row stale | when editing a past session, recalc BOTH the edited day AND today so the rolling 7/28-day windows pick up the change: calculateDailyLoad(userId, editedDate) then calculateDailyLoad(userId, today)
[2026-04-21] | fetched static recommendations from backend; same "take a break" showed for Green/Yellow/Red | backend has no recommendation generator (RecommendationBL only has Create/GetByUser) — the Warnings screen must build the text client-side from loadLevel + acRatio, don't rely on getRecommendationsByUser
[2026-04-21] | weekly chart kept showing high bars after user edited sessions because DailyLoad rows persist their AcuteLoad value from when they were saved (rolling-sum snapshot) — editing ActivityLogs doesn't retroactively update past DailyLoad rows | Refresh on Warnings must recalc ALL 7 days of DailyLoad (loop i=6..0, call calculateDailyLoad(userId, day)), not just today — stale DailyLoad rows are the #1 cause of "status won't change after edits"
[2026-04-21] | Warnings chart sourced bars from DailyLoad.acuteLoad (7-day rolling sum ending on that day) which leaked previous-week sessions into this-week bars | weekly bar chart must aggregate ActivityLogs per-day within the selected week's Sun-Sat window — do NOT use rolling-sum fields for visual per-day bars
[2026-04-21] | Warnings status/AC ratio was pulled from backend DailyLoad rows, which are per-day snapshots that include prior-week sessions in their acute window — status stayed Red for every viewed week even when the bars showed tiny loads | status/AC ratio on the Warnings dashboard is computed client-side from ActivityLogs for the DISPLAYED week: acute = sum(loads in displayed Sun-Sat), chronic = sum(loads in prior 21 days)/3, ratio = acute/chronic. Never read DailyLoad.loadLevel for the per-week view
[2026-04-22] | jumped to soft-delete for ActivityLogs anticipating coach-history access; user later reversed because coach feature will reuse the live trainee dashboard, not a history table | when user mentions a "future feature" as the rationale for added complexity, ask how that future feature will actually work before adding the persistence/schema scaffolding — don't pre-build for assumed designs
[2026-04-22] | wrote `apiClient.post('/api/Users', userData)` in registerUser when the parameter was named `payload` — would have crashed with ReferenceError on signup submit | when you write a function that wraps another call, double-check the argument variable name matches the function signature, especially in axios POST/PUT bodies
[2026-04-22] | left duplicate `<Stack.Screen name="SignUp">` lines in NavigationStack.js after manual edit, breaking the AuthStack with "duplicate screen named" render error | React Navigation's `Stack.Navigator` does not allow two `Screen` children with the same `name` — when adding screens always grep for the existing name first
[2026-04-22] | HomeScreen weekly chart drifted from WarningsDashboardScreen — different aggregation, color thresholds, refresh strategy → bars showed wrong values & wrong colors | when two screens visualize the same data (e.g. weekly load), keep aggregation + color thresholds in a single shared module — don't duplicate getBarColor in both files
[2026-04-22] | StatsScreen ZoomedBarChart hardcoded selected bar to pink (#ff2d6f) regardless of load value | for selection emphasis use opacity/size/border, not a fixed selection color — that destroys the chart's color encoding
[2026-04-22] | HomeScreen used useEffect with [loadData] dependency, never re-fetched after user came back from another tab — deletes on the Health tab didn't propagate | for tab-switched data screens, prefer `useFocusEffect(useCallback(loadFn, deps))` over plain useEffect so the data refreshes when the user returns to the tab

[2026-04-26] | InjuryReportBL.Create validates Severity 1..5 while InjuryReportScreen slider posts 1..10 → submits with severity > 5 silently 400 | whenever you see a numeric range in a UI control, grep the BL/SP for the matching validator before assuming the range is consistent
[2026-04-26] | jumped to "missing IsDeleted filter in SP" as the root cause of recalc 500, but the actual schema never had IsDeleted — real bug was DailyLoadDAL.GetActivityLogsForLoad passing @UserID/@Date when sp_GetActivityLogsForLoad expects @UserID/@StartDate/@EndDate | always read the live SQL Script + the DAL parameter dictionary together; never trust historical comments about "what the SP used to filter on"
[2026-04-26] | Settings save sent only profile fields; backend UpdateUserRequest validates ActivityLevel/ExperienceLevel ∈ [1,3] and BadRequest'd because they were 0 by default | when a screen edits a partial subset of a server-side model that has `range` validators, fetch the full record first and echo unchanged fields back in the payload — never let nullable-with-default fields slip through as 0
[2026-04-26] | HC TotalCaloriesBurned was wired up but it includes BMR — short workouts read 1500-1800 kcal vs Samsung Fit's 200-300 kcal | for workout calories use ActiveCaloriesBurned first; Total only as fallback with a BMR subtraction (~70 kcal/hr) so the number stays sane
[2026-04-26] | HC delete loop: deleting a backend log that originated from HC re-imported it on the next auto-sync because HC still has the record and TrainWise can't write back to HC | Persist a tombstone set in AsyncStorage keyed by normalized startTime; SyncService.deduplicateWorkouts must consult it before posting. [2026-04-26] | WarningsDashboardScreen used "uncoupled" ACWR (chronic = prior 21 days / 3) — caused chronic baseline to swing wildly between adjacent weeks and made the displayed ratio misleading | Use coupled ACWR per Gabbett 2016: chronic = sum(28-day window ENDING at displayed weekEnd, INCLUDING the displayed week) / 4. [2026-04-26] | HomeScreen.buildWeeklyData stored `dayIndex: i` (array position 0..6) and used DAYS[dayIndex] to label, which broke when weekStartDay != Sunday | dayIndex must be the JS day-of-week (date.getDay()), not the array position — only then does DAYS[dayIndex] map to the right label under any week-start setting
[2026-04-26] | shipped a "global Colors swap" theme system but several screens (HomeScreen, StatsScreen, ProfileScreen) had hardcoded hex literals in their styles — light theme appeared half-applied | the ThemeContext only swaps the `Colors` singleton; any screen using `'#xxxxxx'` literals stays dark forever. Before claiming a theme works, grep `'#[0-9a-fA-F]{3,8}'` across `src/screens/**` and `src/components/**` and convert each to `Colors.*` (except semantic traffic-light bar colors)
[2026-04-26] | SettingsScreen called the HTTP `updateUser` but never updated AuthContext, so HomeScreen's "Hello {name}" and ProfileScreen's info card kept showing the cached login snapshot | After any successful PUT /users/{id}, also call `useAuth().updateUser({...})` with the same field names AuthContext normalizes on login (fullName, email, height, weight, etc.) so the in-memory user object + AsyncStorage stay in sync. Rename the HTTP import (e.g. `updateUserApi`) to avoid shadowing the AuthContext method
[2026-04-26] | "global Colors swap" theme didn't apply because `StyleSheet.create({...Colors.x...})` at module level captures hex values at import time — mutating Colors later doesn't update existing stylesheets even with a Fragment-key remount (modules stay cached) | every themed screen MUST move styles inside the component via `useThemedStyles(makeStyles)` where `makeStyles = (Colors) => StyleSheet.create({...})`. The hook is useMemo-keyed on theme and re-runs the factory after applyTheme mutates Colors. Reading Colors.x in JSX/inline-function args still works because the object reference is stable
[2026-04-26] | UserDAL.UpdateUser only passed FullName/BirthYear/Gender/Height/Weight/ActivityLevel/DeviceType/ExperienceLevel — Email, UserName, HealthDeclaration, ConfirmTerms, TermConfirmationDate fell through to sp_UpdateUser's `= NULL` defaults and the SP wrote NULL into the columns on every save | when an SP has nullable-defaulted parameters, the DAL must pass EVERY editable column or the SP silently wipes them. Always grep the SP signature against the param dictionary
[2026-04-26] | UpdateUserRequest DTO was missing UserName + Email fields → SettingsScreen's email change deserialized as null at the controller, BL passed null to DAL, SP wrote NULL → email "deleted" on every save | when a controller does a partial update via DTO, the DTO must include EVERY editable field. Missing properties bind silently to default values (null for strings, 0 for ints) and become hard-to-trace data wipes
[2026-04-26] | Sign-up alert showed "[object Object]" because api.js did `throw new Error(error.response?.data || ...)` and ASP.NET returned a ValidationProblemDetails object (when CreateUserRequest's non-nullable int/byte fields received JSON null from the frontend) | when an axios error's response.data may be either a string or an object, branch on typeof: strings pass through as-is; objects need .errors/.title/.detail extraction or JSON.stringify. Never pass an object to `new Error()` — it stringifies as "[object Object]"
[2026-04-26] | SignUpFinal sent userName: null and birthYear: null to the backend; the BL rejected userName, and ASP.NET 400-failed model binding on int BirthYear because JSON null can't bind to non-nullable int | always provide safe defaults from the frontend for fields the backend declares as non-nullable primitives (e.g. derive userName from email prefix, default birthYear to a sane year). Alternatively make the DTO fields nullable (int? / byte?) — but defaulting at the source is simpler than chasing every binding edge case
[2026-04-26] | every cold app launch red-logged "Sync failed: ... Network Error" twice (once from SyncService, once from useSyncWorkouts) because backend often isn't running when the auto-sync fires | classify network errors as recoverable: pattern-match `/network|timeout|econn|fetch/i` and use console.warn instead of console.error.
